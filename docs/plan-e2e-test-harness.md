# Plan: End-to-End Testing Harness for Cairn Persona Simulation

## Context

Cairn has 12 test profile databases at `tools/test_profiles/{profile_id}/talkingrock.db`.
Each profile is a fully realized persona (acts, scenes, mock emails, mock calendar events,
memories, and a "Your Story" act) generated by the `tools/testgen/` subsystem.

The goal is a harness that:
1. Exercises every profile with a realistic question set derived from its data
2. Captures the full internal pipeline trace per question (classification, tool call, timing, response)
3. Stores results in a queryable format for analysis
4. Does not require Thunderbird running — it reads from the `mock_emails` and `mock_calendar_events`
   tables already in each profile DB

### What we know from codebase research

**Entry points.**
- `agent.respond(user_text, conversation_id=...)` — the full production stack. Creates conversation
  rows, stores messages, runs prompt injection checks, invokes reasoning engine, calls
  `CairnAtomicBridge.process_request()`, returns a `ChatResponse`.
- `CairnAtomicBridge.process_request(user_input, user_id, execute_tool=..., persona_context=...)`
  — one level down. Skips conversation management. Returns `CairnOperationResult` which carries
  `operation`, `verification`, `intent_result`, and `response`.

**Thunderbird dependency.**
`CairnToolHandler._thunderbird` is lazily initialized via `ThunderbirdBridge.auto_detect()`, which
reads the user's actual Thunderbird profile from disk. The `_get_calendar()` and
`_get_upcoming_events()` methods on `CairnToolHandler` check `if self.thunderbird is None` and
return `{"available": False, "events": []}` gracefully. This is the degradation path we will
exploit. The mock data is in `mock_calendar_events` and `mock_emails` inside each profile DB —
we need to intercept the tool call and serve from those tables instead.

**ConsciousnessObserver.**
`ConsciousnessObserver.get_instance()` is a singleton that accumulates events during processing.
Call `observer.start_session()` before each question, then `observer.get_all()` after to collect
the trace. Events carry `event_type`, `timestamp`, `title`, `content`, and `metadata` — including
tool name, tool args, and classification metadata.

**Database.**
`Database(db_path=path)` accepts an explicit path. The profile DBs have full play_db schema
(v13 schema with `acts`, `scenes`, `blocks`, `conversations`, `messages`, etc.) plus
`mock_emails` and `mock_calendar_events`. The `Database` constructor reads
`TALKINGROCK_DATA_DIR` env var when no path is given, so we can point it at each profile
without monkey-patching.

**LLM requirement.**
Classification (`atomic_ops/classifier.py`) and response generation (`cairn/response_generator.py`)
both call Ollama. The harness therefore requires Ollama running. This is unavoidable if we want
real classification behavior — which is the whole point of the test.

**Compute budget reality.**
Each question passes through 2–3 LLM calls:
1. Classification (classifier.py)
2. Optionally a contextual reference resolution call
3. Response generation

At ~4–8 seconds per call on an 8B model:
- 12 profiles × 8 questions × 3 LLM calls × ~6 seconds = ~1,728 seconds ≈ 29 minutes per run
- A 5-question set: ~18 minutes

The harness must be resumable. Results must be persisted question-by-question so a crash
does not lose completed work.

## Approaches Considered

### Approach A — Entry at `agent.respond()` (Recommended)

Invoke the full production stack. Create a `CairnAgent` for each profile, wired to the
profile's DB. Intercept Thunderbird at `CairnToolHandler` by injecting a mock `ThunderbirdBridge`
that reads from the profile's `mock_*` tables.

**Pros:**
- Most realistic path — exercises the exact code the user runs
- ConsciousnessObserver fires naturally
- Conversation context accumulates correctly across questions
- Captures reasoning engine routing decisions
- Tests the full prompt injection check

**Cons:**
- `CairnAgent.__init__()` requires a `Database` instance and an LLM provider, which requires
  knowing the profile's Ollama model config. The profiles all share a local Ollama instance
  so this is manageable.
- Must intercept `ThunderbirdBridge.auto_detect()` to inject mock data

**Mock injection point:** `CairnToolHandler.__init__()` assigns `self._thunderbird = None`
and lazily calls `ThunderbirdBridge.auto_detect()` on first access via the `@property`.
We can set `handler._thunderbird = MockThunderbirdBridge(conn)` immediately after
constructing the handler. To access the handler from the agent, we need to trace through
`agent._cairn_intent_engine` → `intent_engine.tool_handler` (see `cairn/intent_engine.py`).

### Approach B — Entry at `CairnAtomicBridge.process_request()` directly

Bypass `agent.respond()` entirely. Construct a `CairnAtomicBridge` directly, pass a custom
`execute_tool` callable that reads from mock tables for calendar/email tools and delegates to
real `CairnToolHandler` for play/tasks tools.

**Pros:**
- No conversation DB writes — cleaner isolation
- `execute_tool` is passed in, so mock injection is explicit and surgical
- `CairnAtomicBridge.process_request()` is what the agent calls anyway

**Cons:**
- Skips conversation context building (memories, state briefings) — less realistic
- Skips prompt injection check — misses one test dimension
- `CairnAtomicBridge` still needs an `intent_engine` for the LLM references; construction
  is more manual

**Verdict:** Approach B misses conversation context, which is important for realistic response
generation. Approach A is preferred.

### Approach C — HTTP mock server for Thunderbird bridge

Run a small `http.server` on `localhost:19192` that serves mock data in the Thunderbird Bridge
format (the JSON format `ThunderbirdBridgeClient` expects at `/events`, `/health`, etc.).

**Note:** The codebase has *two* Thunderbird integrations. `thunderbird_bridge.py` is an HTTP
client used for write operations (create/update/delete events). `cairn/thunderbird.py` reads
*directly from SQLite* files in the Thunderbird profile directory for read operations
(`list_events`, `get_upcoming_events`). Since calendar queries (read) go through
`cairn/thunderbird.py`, an HTTP mock would not intercept the read path. Approach C does not
solve the problem.

**Verdict:** Ruled out. Does not intercept the actual call path for read operations.

## Recommended Approach: A — Full stack with mock ThunderbirdBridge

### Mock design

`MockThunderbirdBridge(conn: sqlite3.Connection)` implements the same interface as
`ThunderbirdBridge`:

```
list_events(start, end) → list[CalendarEvent]
get_upcoming_events(hours, limit) → list[CalendarEvent]
search_contacts(query, limit) → list[Contact]
get_status() → dict
```

It reads from `mock_calendar_events` (columns: `id`, `title`, `start_time`, `end_time`,
`location`, `description`, `calendar_name`, `all_day`, `recurrence_rule`, `attendees`) and
`mock_emails` (columns: `id`, `message_id`, `subject`, `sender`, `recipients`, `date`, `body`,
`is_read`, `folder`, `has_attachment`, `importance`).

Emails are not a standard Thunderbird `list_events` return type, but email queries get routed
to a different tool (`cairn_surface_attention` or `PERSONAL`/`CONVERSATION` modes). For the
specific tools that hit Thunderbird, only calendar data needs mocking.

### Question generation

Questions are generated per-persona at runtime by reading the profile DB and templating from
actual data. Generation is deterministic (seeded random where needed) so re-runs produce the
same question set.

**8 questions per persona covering all 8 query types:**

| # | Type | Example template |
|---|------|-----------------|
| 1 | Calendar query | `"What's on my calendar today?"` |
| 2 | Email query | `"Any urgent emails I should know about?"` |
| 3 | Project query | `"What's the status of {act.title}?"` (filled from DB) |
| 4 | Task query | `"What should I work on next?"` |
| 5 | Personal query | `"Tell me about myself"` |
| 6 | Vague/low-quality | `"stuff"`, `"help"`, `"???"` |
| 7 | Multi-intent | `"Check my calendar and tell me about {act.title}"` |
| 8 | Off-topic | `"What's the weather like in Tokyo?"` |

Templates 3 and 7 are filled from actual act titles in the profile, making them feel authentic.

## Implementation Steps

### Step 1 — Create the harness directory

```

├── question_generator.py     # Per-persona question set generation
├── runner.py                 # Main test runner (profile loop, question loop, LLM calls)
├── recorder.py               # Writes results to harness.db and JSONL
├── analysis.py               # SQL/pandas helper queries for post-run analysis
└── README.md
```

**File:** `tools/run_harness.py` — top-level script, `python3 tools/run_harness.py [profile_id]`

### Step 2 — Implement `mock_thunderbird.py`

Implement `MockThunderbirdBridge`:
- Constructor takes `conn: sqlite3.Connection`
- `list_events(start=None, end=None)` — queries `mock_calendar_events` on `start_time`/`end_time`
  and maps rows to `CalendarEvent` dataclasses (from `cairn/thunderbird.py`)
- `get_upcoming_events(hours=24, limit=10)` — filters `mock_calendar_events` where
  `start_time` is within `now + hours`
- `search_contacts(query, limit)` — returns empty list (no mock contact data in profiles)
- `get_status()` — returns `{"available": True, "source": "mock"}`
- `is_available()` — returns `True`

**Injection point:** After constructing the agent, navigate to the tool handler and set:
```python
agent._cairn_intent_engine.tool_handler._thunderbird = MockThunderbirdBridge(conn)
```
This bypasses `auto_detect()` without monkey-patching at module level.

To find the correct attribute path, verify against `src/cairn/cairn/intent_engine.py` — look
for where `CairnToolHandler` is instantiated and what attribute it lives on.

### Step 3 — Implement `question_generator.py`

```python
def generate_questions(profile_id: str, conn: sqlite3.Connection) -> list[dict]:
    """Generate 8 questions for a profile using actual DB data."""
```

Reads:
- `acts` table for act titles (filter out `system_role IN ('your_story', 'archived_conversations')`)
- `mock_calendar_events` for today's events (for calendar query flavor)
- `mock_emails` for unread email subject (for email query flavor)

Returns a list of dicts:
```python
{
    "question_id": "q-{profile_id}-{n}",
    "query_type": "calendar|email|project|task|personal|vague|multi_intent|off_topic",
    "text": "...",
    "profile_id": profile_id,
}
```

### Step 4 — Implement `recorder.py`

Creates `tools/harness_results/harness.db` with two tables:

```sql
CREATE TABLE run_questions (
    run_id TEXT,
    question_id TEXT PRIMARY KEY,
    profile_id TEXT,
    query_type TEXT,
    question_text TEXT,
    status TEXT DEFAULT 'pending',  -- pending|complete|error
    started_at TEXT,
    completed_at TEXT
);

CREATE TABLE run_results (
    question_id TEXT PRIMARY KEY,
    run_id TEXT,
    profile_id TEXT,
    query_type TEXT,
    question_text TEXT,
    response_text TEXT,
    classification_domain TEXT,
    classification_action TEXT,
    classification_confident INTEGER,
    tool_called TEXT,
    tool_args TEXT,        -- JSON
    tool_result_preview TEXT,
    verification_passed INTEGER,
    needs_approval INTEGER,
    total_ms INTEGER,
    consciousness_events TEXT,  -- JSON array of event summaries
    error TEXT,
    completed_at TEXT
);
```

Also writes one JSONL file per run (`harness_results/{run_id}.jsonl`) with the full
`consciousness_events` array per result — this is the rich event-by-event trace.

**Resumability:** Before running a question, the runner checks `run_questions` for
`status='complete'`. If found, it skips. This means a crash mid-run can be resumed
by re-running with the same `run_id`.

### Step 5 — Implement `runner.py`

```python
def run_profile(profile_id: str, run_id: str, recorder: Recorder) -> None:
    """Run all questions for one profile."""
```

Flow per question:
1. Check recorder — skip if already complete
2. Call `observer.start_session()`
3. Record start time
4. Call `agent.respond(question_text, conversation_id=conversation_id)`
5. Collect `observer.get_all()` and `observer.end_session()`
6. Extract classification/tool/timing from `CairnOperationResult` (available via
   `agent._last_result` if we patch, or by parsing consciousness events)
7. Record result

**Getting structured result data from agent.respond():** `agent.respond()` returns a
`ChatResponse` which contains the response text but not the full `CairnOperationResult`.
To get the rich metadata (classification domain, tool name, tool args, verification status),
we have two sub-options:

- **Sub-option A (preferred):** Extract from ConsciousnessObserver events. The observer
  captures `INTENT_EXTRACTED` events with `category`, `action`, `confidence` in metadata,
  and `TOOL_CALL_START` events with `tool` in metadata. This is already there.
- **Sub-option B:** Bypass `agent.respond()` and call `CairnAtomicBridge.process_request()`
  directly, which returns `CairnOperationResult` with full classification. We lose
  conversation context building but gain structured result access.

**Recommendation:** Use Sub-option A for full realism. Parse consciousness events after each
turn. The events provide enough structured data:
- `INTENT_EXTRACTED` metadata → `category`, `action`, `confidence`
- `TOOL_CALL_START` metadata → `tool`
- `TOOL_CALL_COMPLETE` metadata → `success`
- `RESPONSE_READY` → final, total event count gives a processing depth metric

### Step 6 — Implement `analysis.py`

Provide helper queries that print to stdout or return DataFrames:

```python
def query_by_type(db_path) -> pd.DataFrame:
    """Classification accuracy by query_type (did 'calendar' queries use calendar tool?)"""

def query_failures(db_path) -> pd.DataFrame:
    """All questions where error is non-null or verification_passed = 0"""

def query_tool_dispatch(db_path) -> pd.DataFrame:
    """Pivot: query_type × tool_called counts"""

def query_timing(db_path) -> pd.DataFrame:
    """P50/P95 response time by query_type and profile"""

def query_per_profile(db_path, profile_id) -> pd.DataFrame:
    """Full trace for one profile"""
```

These are read-only queries against `harness.db`.

### Step 7 — Write `tools/run_harness.py`

Top-level script:
```
python3 tools/run_harness.py                    # all 12 profiles
python3 tools/run_harness.py aisha_osman        # one profile
python3 tools/run_harness.py --resume run-abc   # resume a prior run
```

Outputs:
- Live progress: `[aisha_osman] q1/8 calendar — 4.2s OK`
- Summary table at end: per-profile success rate, avg timing, tool dispatch breakdown

## Risks and Mitigations

### Risk 1 — `ThunderbirdBridge` injection path may differ from assumed

**Detail:** The injection assumes `agent._cairn_intent_engine.tool_handler._thunderbird`. The
actual attribute chain depends on `intent_engine.py` internals not fully read during research.

**Mitigation:** The implementer must read `src/cairn/cairn/intent_engine.py` before coding
the injection. The fallback is to monkey-patch `ThunderbirdBridge.auto_detect` at module level
with `monkeypatch`-style replacement before the agent is constructed. Less elegant but reliable.

### Risk 2 — Profile DB schema may not match expected `CalendarEvent` dataclass fields

**Detail:** `mock_calendar_events` has `start_time`/`end_time` columns; `CalendarEvent`
(from `cairn/thunderbird.py`) may use different field names (`start`/`end`, `datetime` objects).

**Mitigation:** The implementer must read `CalendarEvent`'s definition in `cairn/thunderbird.py`
before writing `MockThunderbirdBridge`. Field mapping should be explicit, not assumed.

### Risk 3 — Conversation DB writes to profile DB pollute test data

**Detail:** `agent.respond()` writes `conversations` and `messages` rows to the profile DB.
Repeated runs accumulate this data, potentially affecting subsequent question context.

**Mitigation:** Use a separate ephemeral DB for conversation state. Either:
- Pass a `:memory:` or temp-dir DB as the `conversations`/`messages` store (requires the
  ability to have two DBs open — verify agent supports this), or
- Copy the profile DB to a temp path before each run and discard after

Simplest: copy the profile DB to a temp dir at run start. This preserves the profile DB
immutably and allows full cleanup.

### Risk 4 — ConsciousnessObserver is a singleton; parallel profile runs would collide

**Detail:** `ConsciousnessObserver._instance` is a process-level singleton. Running profiles
in parallel threads would interleave events.

**Mitigation:** Run profiles sequentially. Do not parallelize. The compute bottleneck is
Ollama anyway (single-threaded inference), so parallelism buys nothing.

### Risk 5 — Ollama not running or wrong model

**Detail:** The provider factory reads `ollama_model` from the profile DB's `app_state` table.
If the profiles were generated without setting this, it will use the default model, which may
not be installed.

**Mitigation:** The harness should check Ollama availability at startup (call
`check_provider_health(db)`), print the model name being used, and fail fast if the model is
not available rather than silently returning empty responses.

### Risk 6 — play_fs.py reads from `os.environ["TALKINGROCK_DATA_DIR"]`

**Detail:** `play_fs.py` and related modules may use a global path to locate the DB.
If they do, tool calls that touch play data (acts, scenes) may read the user's live DB
rather than the profile DB.

**Mitigation:** Set `TALKINGROCK_DATA_DIR` to the profile's directory before constructing the
agent. The `Database.__init__()` code confirms it reads this env var. The harness must set
this before any imports that touch `play_fs`. Implementer must verify no module-level path
caching exists that would ignore this.

## Testing Strategy

The harness is primarily a testing tool, not tested code. However:

1. **Smoke test:** `tools/run_harness.py aisha_osman --dry-run` — generates questions, prints
   them, does not invoke Ollama. Verifies DB reads, mock injection, and question generation.

2. **Single-question test:** Run one profile with `--limit 1` (one question). Verify JSONL
   output and harness.db record contain expected fields.

3. **Mock verification:** After running a calendar question, confirm the `tool_called` column
   in `harness.db` is `cairn_get_calendar` and the tool result shows mock data, not
   `{"available": False, "events": []}`.

4. **Full run validation:** After a complete 12-profile run, run `analysis.py` queries. Expect:
   - Calendar questions have `classification_domain = 'calendar'` in > 80% of cases
   - Off-topic questions have no `tool_called` value
   - Vague questions result in `classification_confident = 0` in most cases

## Confidence Assessment

**Confidence: High (85%).**

The key architectural facts are confirmed:
- ThunderbirdBridge is lazily initialized and injectable
- ConsciousnessObserver is the natural trace capture mechanism and is already wired in the pipeline
- Profile DBs have real act/scene data suitable for question templating
- Database accepts explicit path or env-var override

**Uncertainties requiring implementer verification:**
- Exact attribute path to `CairnToolHandler._thunderbird` from a constructed agent (requires
  reading `intent_engine.py` in full)
- Whether `play_fs.py` uses a cached path that would ignore `TALKINGROCK_DATA_DIR` when set
  after import (requires tracing `play_fs.py`)
- Whether the `CalendarEvent` dataclass fields match the `mock_calendar_events` column names
  one-for-one (requires reading the dataclass definition in `cairn/thunderbird.py`)

These are localized uncertainties. None are blockers — each has a stated fallback in the Risks section.
